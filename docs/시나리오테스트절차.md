# 시나리오 테스트 절차

> **목적**: 스프린트 1~6 범위의 주요 사용자 흐름을 한 번에 검증할 때 따르는 **수동 절차**입니다.  
> **참조**: [제품백로그.md](./제품백로그.md), [스프린트플래닝.md](./스프린트플래닝.md), [개발환경설정.md](./개발환경설정.md)

## 1. 문서 범위

| 포함 | 제외(별도 계획) |
|------|----------------|
| 웹(Next.js) 화면·API·실시간(STOMP) 연동 흐름 | 모바일 앱 전용(FCM 푸시 토큰 등) |
| 제품 백로그 Epic 2~6 중심 | 관리자·신고·공지 등 백로그 미포함 기능 |

### 1.1 역할 분담 (Bearer access token 2개 제공 전제)

카카오(A)·구글(B) 등으로 발급한 **JWT 2개**를 채팅 등으로 넘겨주고 협업할 때의 역할 구분입니다.  
**보안**: 토큰은 저장소·이슈에 넣지 말고, 가능하면 **개발용 계정·짧은 수명**만 사용하고 테스트 후 **재발급·폐기**합니다.

| 구분 | 사용자(직접) | AI·에이전트(협업 시 가능한 범위) |
|------|-------------|----------------------------------|
| **토큰 확보·수명** | OAuth 로그인으로 A/B 토큰 발급, 만료 시 재로그인·재전달 | 대신 로그인 불가; 만료되면 새 토큰 요청 |
| **시나리오 1** UI | 최초 가입 폼, 프로필 화면, 이미지 업로드, 타인 프로필 라우팅 확인 | REST `GET /users/me` 등으로 **데이터 일치** 보조 검증 가능 |
| **시나리오 2~4** REST 중심 | (선택) UI로 생성·목록 클릭 | 토큰으로 `curl`/스크립트 호출해 **매칭·참여·후기·패널티 API** 응답·상태코드 검증, 응답 JSON 해석·기대값 대조 |
| **시나리오 5 실시간** | 브라우저 두 개로 채팅방 입장, STOMP 수신·배지, `5-7` 끊김/재접속·Network에서 `afterId` 확인 | 브라우저 대신 할 수 없음. STOMP는 별도 클라이언트 스크립트로 **연결·수신 여부**만 원천 검증하는 수준은 가능(웹 UX와 동일 검증은 아님) |
| **시나리오 6-5** | 실제 토스트·문구가 읽기 쉬운지 육안 확인 | API `code`와 앱의 `showApiErrorToast` 매핑 표를 **코드 기준으로 대조**·누락 코드 제안 |
| **Gate·헬스** | 브라우저 접속 | `curl` 헬스·공개 엔드포인트 확인 |
| **3-7 시간 초과** | 15분(또는 설정값) 실제 대기·UI 확인 | 장시간 대기 대행 불가; 스케줄러 로그·DB 상태는 사용자가 캡처해 주면 해석 도움 |
| **2.1.1 Redis 2인스턴스** | 서버 2대 기동·포트 분리·프론트 URL 스위치 | 동일 |
| **판정·이력** | 최종 통과/실패, 스크린샷, 이슈 트래킹 | [8절 이슈 추적표](#8-시나리오-테스트-결과-기록이슈-추적표) 초안 작성·API 로그 문장화 도움 |
| **자동화 확장** | Playwright에 토큰 주입(`localStorage`) 시나리오는 로컬에서 실행 | 테스트 코드·`storageState` 패턴 초안 작성 |

요약: **화면·실시간·장시간·OAuth는 사용자**, **REST API 검증·응답 분석·스크립트/E2E 초안은 토큰만 있으면 AI와 나란히 진행**하기 좋습니다.

## 2. 사전 준비

### 2.1 환경

1. 백엔드(Spring Boot) 기동, DB·Redis 등 백엔드 문서 요구사항 충족.
2. 프론트: `.env.local`에 `NEXT_PUBLIC_API_BASE_URL` 등 설정 ([개발환경설정.md](./개발환경설정.md) 참고).
3. 로컬에서 `pnpm dev`(또는 `npm run dev`)로 웹 서버 실행(기본 `http://localhost:3000`).
4. OAuth(카카오/네이버/구글/애플) 리다이렉트 URI가 **로컬·백엔드 양쪽**에 등록되어 있는지 확인.

### 2.1.1 (선택·심화) Redis 릴레이·다중 인스턴스

**목적**: Redis Pub/Sub 릴레이를 켠 환경에서 **수평 확장 시 채팅·실시간 메시지가 인스턴스 간에 전달**되는지 증명합니다.

**전제**: 백엔드가 동일 Redis를 바라보고, STOMP 릴레이 관련 설정이 문서대로 켜져 있음 (`STOMP_REDIS_RELAY_ENABLED` 등, `common-docs/api/Sprint6-API.md`·백엔드 README 참고).

| 순서 | 수행 | 기대 결과 |
|------|------|-----------|
| R-1 | 로컬에서 API 서버를 **두 개** 기동(예: 포트 `8080`, `8081`), 프론트를 두 브라우저에서 각각 `NEXT_PUBLIC_API_BASE_URL`을 **서로 다른 포트**로 두고 빌드·실행하거나, 동일 오리진 프록시로 라우팅만 분기할 수 있는 환경을 구성 | 동일 채팅방 roomId로 각각 접속 가능 |
| R-2 | 사용자 A는 `8080`에 붙인 웹, B는 `8081`에 붙인 웹에서 **동일 채팅방** 입장 | 양쪽 SockJS가 각자 인스턴스에 연결됨 |
| R-3 | A가 메시지 전송 | B 화면에 수신(릴레이 미작동 시 한쪽에서만 보이는 현상으로 실패 판정) |

> 인프라 제약으로 한 인스턴스만 띄울 수 있으면 본 절은 **보류**하고, 스테이징/운영에서 동일 절차로 재검증합니다.

### 2.2 계정·브라우저

- **최소 2명 분** 권장: 방장(A), 참여자(B). 서로 다른 브라우저 또는 시크릿 창으로 동시에 검증하면 알림·채팅 실시간 확인이 수월합니다.
- 브라우저 개발자 도구 **Network** 탭으로 401/403, SockJS(`/ws-chat`) 협상 실패 여부를 병행 확인할 수 있습니다.

### 2.3 결과 기록

각 시나리오 끝에 **통과 / 실패 / 보류**와 실패 시 **재현 순서·스크린샷·콘솔·응답 본문**을 남깁니다.  
백엔드-only 이슈는 API 응답 `code`·HTTP 상태를 함께 적습니다.

**권장**: 아래 [8. 시나리오 테스트 결과 기록(이슈 추적표)](#8-시나리오-테스트-결과-기록이슈-추적표) 양식으로 한 장에 모아 두면 회귀·보고·이력서용 트러블슈팅 정리에 유리합니다.

## 4. 시나리오별 절차

### 시나리오 1 — 회원가입·로그인·프로필 (Epic 2)

**전제**: 소셜 계정 1개 이상.

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 1-1 | `/login`에서 제공되는 제공자 중 하나로 로그인 | 콜백 후 JWT 저장, 서비스 화면 진입 | ㅇ
| 1-2 | 최초 가입 시 `/signup`(또는 안내 경로)에서 닉네임·관심 지역 등 입력 | 제출 후 메인/프로필 이용 가능 | ㅇ
| 1-3 | `/profile/me`에서 프로필 조회·수정 | 저장 후 재조회 시 반영 | ㅇ
| 1-4 | 프로필 이미지 업로드(가능한 경우) | 미리보기·URL 반영 | ㅇ
| 1-5 | `/profile/[userId]`로 **다른 사용자** 프로필 조회 | 공개 정보·후기/패널티 진입 링크 등 백로그와 일치 | ㅇ

### 시나리오 2 — 매칭 생성·목록·상세 (Epic 3, 8)

**전제**: 사용자 A 로그인.

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 2-1 | `/matching/create`에서 제목·일시·장소·인원·급수·코트비·이미지(선택) 입력 후 생성 | 상세로 이동 또는 목록에 노출. **목록 카드**에 업로드한 매칭 이미지(`imageUrl`) 썸네일·**모집 상태 배지**(모집중/마감임박 등)가 보일 것 |
| 2-2 | `/matching`에서 관심 지역·날짜·**급수**·「모집 중만」 필터 적용 | API 쿼리와 화면에 반영. **시간대·성별 필터는 제품 범위에서 제외**(UI 미노출). 급수는 `targetLevels` 토큰 **완전 일치** 규칙으로 필터될 것([Sprint2-API.md](../common-docs/api/Sprint2-API.md) `level` 파라미터 설명 참고) |
| 2-3 | 카드 클릭으로 `/matching/[id]` 상세 | 방장 정보, 모집 상태, 확정/대기 인원 요약 표시 |
| 2-4 | (방장 A) `/matching/[id]/edit`에서 수정 | 저장 후 상세에 반영 |

#### 시나리오 2 — 테스트에서 나온 이슈·대응 (참고)

| 주제 | 원인 추정 / 조치 |
|------|------------------|
| 목록에 매칭 사진 없음 | 프론트: 목록 카드에 `imageUrl` 표시(반영됨). 여전히 비면 `GET /api/matches` 응답 `content[].imageUrl` 누락 여부를 Network로 확인 → **백엔드 목록 DTO에 `imageUrl` 포함** 필요 |
| 목록에 상태 안 보임 | 프론트: `status` 없으면 **「상태 미제공」**(반영됨). 실서비스에서는 `content[].status` 필수 권장. 누락 시 **목록 조회 API 응답에 `status` 추가** |
| B 필터인데 초심·D·C 방만 있는데도 목록에 잡힘 | 백엔드: `level=B`를 `LIKE '%B%'` 등으로 걸면 **`BEGINNER`가 B로 오탐**될 수 있음. **쉼표 분리 후 토큰 전체 일치**로 필터하도록 수정. 명세는 `common-docs/api/Sprint2-API.md`에 보강함 |
| A 필터 시 위 방들이 안 보임 | `targetLevels`에 `A`가 없으면 제외되는 동작은 정상 |

### 시나리오 3 — 참여·대기·수락·거절·취소·대기 승인 (Epic 4)

**전제**: 사용자 A(방장), B(참여자). 모집 중 매칭 1건.

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 3-1 | B가 상세에서 참여 신청(멘트 입력) | 신청 후 상태 표시(대기/신청 완료 등) |
| 3-2 | A에게 알림 또는 신청 목록 갱신 | **매칭 상세** 헤더의 알림 아이콘으로 `/notifications` 이동 가능, STOMP·배지는 기존과 동일. 방장은 **같은 상세 화면의 신청 목록**에서도 신청을 확인·수락·거절할 수 있음 |
| 3-3 | A가 B **수락** | B가 확정 참여자로 표시 |
| 3-4 | (정원 초과 상황 구성 시) B를 대기 신청 | 대기 순번 표시 |
| 3-5 | A가 확정자 **거절** 또는 B가 **참여 취소** | 상태·정원·대기열이 문서와 일치하게 갱신 |
| 3-6 | 확정자 취소 후 **대기 1번에게 기회** 부여 | 백엔드 정책대로 오퍼/승인 UI 노출(프로젝트 화면에 맞춰 확인) |
| 3-7 | **(엣지·시간)** 대기 1번에게 기회가 갔을 때, 정책상 제한 시간(예: **15분**) 동안 아무 반응이 없으면 **대기 2번에게 기회가 이월**되는가 | `@Scheduled`·`ShedLock`(또는 팀에서 쓰는 스케줄 중복 방지)·Redis TTL 등 **운영 시간 축**으로 설계된 로직이 실제로 동작하는지 확인. 타임아웃 분은 백엔드 명세·설정 값에 맞출 것 |
| 3-8 | **(선택·동시성)** 서로 다른 두 참여자가 **동시에** 신청(더블 클릭·동시 탭)했을 때 | 정원·대기열·알림이 데이터 정합성을 깨지 않고 한 번만 반영되는지 확인(아래 [3-8 동시 신청 및 API 검증](#38-동시-신청-및-api-검증) 참고) |

#### 3-7·3-8 참고 (수동 테스트 메모)

- 오퍼 유효 기간 동안 대기 1번이 아무 조치도 하지 않으면, 방장 화면·정책에 따라 신청 목록/알림 표시와 **재수락 가능 여부**가 달라질 수 있음. 제품 의도와 백엔드 스펙을 맞춘 뒤 UI를 조정한다.

#### 3-8 동시 신청 및 API 검증

브라우저로 두 사람이 **동일 밀리초**에 누르기는 어렵습니다. 아래 중 하나로 대체합니다.

| 방법 | 설명 |
|------|------|
| **API 연속 호출** | `POST`(또는 팀이 쓰는 참여 신청 API)를 **두 개의 Bearer**(서로 다른 유저)로, `Promise.all`·`&` 백그라운드·짧은 간격 `curl` 등으로 **거의 동시에** 여러 번 호출. 응답 HTTP·본문·이후 `GET` 상세로 정원·대기열이 한 번만 반영되는지 확인 |
| **스크립트** | 저장소에 `scripts/` 등에 Node 또는 shell 예시를 두고, 로컬 API URL·`matchId`·토큰만 넣어 실행(토큰은 커밋 금지) |
| **에이전트 협업** | API가 떠 있고 **두 계정 JWT**를 일회성으로 넘겨주면, 터미널에서 동일 시나리오용 `curl`을 대신 실행·응답 요약은 가능(실제 서버·DB·네트워크에 따라 결과는 당사자 환경에서 최종 검증) |

**주의**: 부하·데이터 오염을 막기 위해 **개발 DB**에서만 수행하고, 테스트 후 필요 시 참여 행 정리.

### 시나리오 4 — 후기·패널티 (Epic 5)

**전제**: 매칭이 종료 가능한 상태(백엔드 규칙: 예: `FINISHED`).

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 4-1 | `/matching/[id]/review`(또는 안내 경로)에서 후기 작성 | 3단계 평가·해시태그 등 입력 후 성공 | ㅇ
| 4-2 | `/profile/[userId]/reviews` 또는 `/profile/me/reviews`에서 후기 목록 | 방금 작성분 노출 | ㅇ
| 4-3 | (방장) 노쇼/지각 등 패널티 UI에서 체크 | API 성공, 대상 프로필에 반영 | ㅇ
| 4-4 | `/profile/.../penalties`에서 패널티 이력 확인 | 항목·일시 표시 | ㅇ

### 시나리오 5 — 채팅·실시간·알림 (Epic 6)

**전제**: 동일 매칭에 확정된 사용자 2명 이상, 채팅방 생성됨.

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 5-1 | 확정 후 `/chat` 목록에 방 표시 또는 `/chat/from-match/[matchId]` 진입 | 채팅방 ID 확인 가능 | ㅇ
| 5-2 | `/chat/[roomId]`에서 상단 공지(일시·장소·코트비 등) 표시 | 백로그 B6-3와 일치 | ㅇ
| 5-3 | A가 메시지 전송 | B 화면에 수신(STOMP 연결 시 실시간, 끊김 시 폴링/재연결 정책 확인) | ㅇ
| 5-4 | 연결 상태 표시(헤더 배지 등) | 끊김·재연결 시 사용자 안내 | ㅇ
| 5-5 | 매칭 **종료·취소** 시 시스템 메시지·입력 비활성 | 서버 정책과 UI 일치 | ㅇ
| 5-6 | 신청·대기열 관련 이벤트 시 `/notifications` 목록·미읽음 | STOMP 수신 시 목록/배지 갱신(백엔드 이벤트 필요) |
| 5-7 | **(갭 복구)** 채팅 중 **탭 종료** 또는 **네트워크 차단**으로 STOMP가 끊긴 뒤, 상대방이 그 사이 **새 메시지를 보내고**, 이후 끊긴 쪽이 **다시 접속** | [Sprint6-할일.md](./Sprint6-할일.md)에서 합의한 대로, 재연결 핸드셰이크 직후 **`GET .../messages?afterId=...`(또는 동등한 갭 보정)** 이 호출되고, 놓친 메시지가 목록에 **REST 응답과 병합**되어 나타나는지 Network 탭에서 확인. WS 유지 중에는 `afterId` 폴링이 멈추는 정책이면 그 전후도 함께 본다 |

### 시나리오 6 — 보안·예외·비기능 (Epic 7, 스프린트 6)

| 단계 | 수행 | 기대 결과 |
|------|------|-----------|
| 6-1 | 로그아웃 또는 `localStorage`에서 `accessToken` 삭제 후 보호된 URL 접근 | 로그인 유도 | ㅇ
| 6-2 | 잘못된 `roomId`로 `/chat/999999999` 접근 | 친절한 오류 메시지, 목록으로 복귀 가능 | ㅇ
| 6-3 | 네트워크 오프라인 후 복구 | 재요청·재연결 또는 안내 토스트 |
| 6-4 | (선택) 개발 모드 `dev/stomp` 페이지 | SockJS URL·연결 로그로 STOMP 점검(Sprint 6 문서 참고) |
| 6-5 | **(에러 UX)** API·STOMP 비즈니스 오류 시 응답 본문의 **`code`** 가 사용자에게 **이해 가능한 문구**로 바뀌는가 | 예: 매칭이 이미 종료된 상태에서 메시지 전송 시 `INVALID_MATCH_STATUS` 등이 `"종료된 매칭에는 메시지를 보낼 수 없습니다"` 같은 **토스트/인라인 안내**로 노출. 반대로 raw JSON·내부 코드만 보이면 실패로 기록 |

## 5. 자동화 스모크(보조)

저장소에 Playwright 스모크가 있으면 시나리오 전 **회귀 상시**에 포함할 수 있습니다.

```bash
pnpm exec playwright install   # 최초 1회
pnpm run test:e2e
```

현재 스모크는 로그인 페이지·채팅 비로그인 게이트 등 **일부 경로**만 검사합니다. 전 구간은 본 문서의 수동 시나리오로 보완합니다.

## 6. 완료 기준 제안

- 시나리오 **1~5**를 방장/참여자 역할로 **각각 최소 1회** 통과.
- 시나리오 **6**에서 인증·오류 메시지 매핑(6-5)·오류 UX에 치명적 결함 없음.
- **심화**(3-7 갭·5-7 재동기화·2.1.1 Redis 릴레이)는 인력·환경 여유 시 수행; 실패 시 원인 분석 자체가 포트폴리오 스토리로 가치 있음.
- 발견 이슈는 백엔드/프론트 라벨과 함께 이슈 트래커에 등록.
